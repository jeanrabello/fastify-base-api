# Entities Development Guide

## Overview

Entities represent the core domain objects of the application. They define the structure of business objects and serve as the data models for database collections and API responses.

## Architecture Pattern

### Base Entity Structure

Entities are TypeScript interfaces that define the shape of domain objects:

```typescript
export interface {EntityName} {
  id?: string;           // Optional, generated by database
  // Domain-specific fields
  name: string;
  email: string;
  status: string;
  // Audit fields
  createdAt?: Date;      // Auto-generated
  updatedAt?: Date;      // Auto-updated
}
```

## Entity Template

```typescript
export interface {EntityName} {
  // Primary identifier
  id?: string;

  // Required business fields
  requiredField1: string;
  requiredField2: number;

  // Optional business fields
  optionalField1?: string;
  optionalField2?: boolean;

  // Enumerated fields
  status: 'active' | 'inactive' | 'pending';

  // Relationships (reference by ID)
  parentId?: string;
  categoryIds?: string[];

  // Sensitive fields (excluded from responses)
  password?: string;
  secretKey?: string;

  // Audit fields
  createdAt?: Date;
  updatedAt?: Date;
  createdBy?: string;
  updatedBy?: string;
}
```

## Real-World Example: User Entity

```typescript
export interface User {
```

## Design Principles

### 1. Domain Focus

- Entities represent business concepts, not technical artifacts
- Use domain terminology and concepts
- Focus on what the entity represents, not how it's stored

### 2. Immutability Consideration

- Define fields as readonly when they shouldn't change after creation
- Use optional fields for generated values (id, timestamps)
- Consider using interfaces for flexibility

### 3. Type Safety

- Use specific types instead of generic ones
- Leverage TypeScript's type system for validation
- Use union types for enumerated values

### 4. Separation of Concerns

- Keep entities pure data structures
- Avoid mixing business logic with data structure
- No dependencies on external libraries

## Naming Conventions

### Entity Names

- Use PascalCase for entity names
- Use singular form (User, not Users)
- Use clear, descriptive names from domain language

### Field Names

- Use camelCase for field names
- Use descriptive names that reflect business meaning
- Avoid abbreviations unless they're domain standard

### Examples

```typescript
// Good naming
User, Product, Order, OrderItem
username, firstName, lastName
isActive, hasPermission, canEdit
createdAt, updatedAt, lastLoginAt

// Avoid
Users, Products (plural)
usr, fname, lname (abbreviated)
active, permission, edit (unclear boolean context)
created, updated (incomplete)
```

## Entity Relationships

### One-to-Many

```typescript
// Parent entity stores no reference
export interface Category {
  id?: string;
  name: string;
  description?: string;
}

// Child entity references parent
export interface Product {
  id?: string;
  name: string;
  categoryId: string; // Reference to Category
  price: number;
}
```

### Many-to-Many

```typescript
// Use junction/bridge pattern
export interface User {
  id?: string;
  username: string;
  roleIds: string[]; // Array of role IDs
}

export interface Role {
  id?: string;
  name: string;
  permissions: string[];
}

// Alternative: explicit junction entity
export interface UserRole {
  userId: string;
  roleId: string;
  assignedAt: Date;
  assignedBy: string;
}
```

### Embedded Objects

```typescript
// For closely related data
export interface User {
  id?: string;
  username: string;
  profile: {
    firstName: string;
    lastName: string;
    avatar?: string;
    bio?: string;
  };
  preferences: {
    theme: "light" | "dark";
    language: string;
    notifications: boolean;
  };
}
```

## Database Considerations

### MongoDB Integration

```typescript
// Entity definition
export interface User {
  id?: string;           // Maps to _id in MongoDB
  username: string;
  email: string;
  createdAt?: Date;
}

// Repository mapping
private mapToResponse(mongoDoc: User & { _id: ObjectId }): Partial<User> {
  return {
    id: mongoDoc._id.toString(),  // Convert ObjectId to string
    username: mongoDoc.username,
    email: mongoDoc.email,
    createdAt: mongoDoc.createdAt,
    // Exclude password and other sensitive fields
  };
}
```

### Index Considerations

```typescript
// Design entities with query patterns in mind
```

## Security Considerations

### Sensitive Data Handling

```typescript
export interface User {
  id?: string;
  username: string;
  email: string;
  password: string; // Never returned in API responses
  passwordResetToken?: string; // Internal use only
  emailVerificationToken?: string; // Internal use only
  createdAt?: Date;
  updatedAt?: Date;
}

// Response model excludes sensitive fields
export interface UserResponseModel {
  id: string;
  username: string;
  email: string;
  createdAt: Date;
  updatedAt: Date;
  // password and tokens excluded
}
```

### Data Classification

```typescript
// Mark sensitive fields clearly
export interface UserProfile {
  // Public data
  id?: string;
  username: string;
  displayName: string;

  // Private data (owner only)
  email: string;
  phone?: string;

  // Sensitive data (encrypted)
  ssn?: string; // PII - encrypt at rest
  bankAccount?: string; // PCI - encrypt at rest

  // Internal data (system only)
  passwordHash?: string; // Never expose
  internalNotes?: string; // Admin only
}
```

### Type Guards

```typescript
// Validate entity structure at runtime
export function isUser(obj: any): obj is User {
  return (
    typeof obj === "object" &&
    typeof obj.username === "string" &&
    typeof obj.email === "string" &&
    typeof obj.password === "string"
  );
}

// Validate entity arrays
export function isUserArray(arr: any[]): arr is User[] {
  return Array.isArray(arr) && arr.every(isUser);
}
```

## Best Practices

### 1. Keep It Simple

- Entities should be plain data structures
- Avoid complex nested objects when possible
- Use composition over complex inheritance

### 2. Domain-Driven Design

- Use terminology from the business domain
- Entities should make sense to domain experts
- Avoid technical implementation details

### 3. Evolution and Versioning

- Design entities to be extensible
- Use optional fields for new features
- Consider backward compatibility

### 4. Documentation

- Document complex field relationships
- Explain business rules and constraints
- Provide examples of valid data

## Migration Considerations

When evolving entities:

```typescript
// Version 1
export interface User {
  id?: string;
  username: string;
  email: string;
}

// Version 2 - Add optional field
export interface User {
  id?: string;
  username: string;
  email: string;
  firstName?: string; // New optional field
}

// Version 3 - Restructure (requires migration)
export interface User {
  id?: string;
  username: string;
  email: string;
  profile: {
    // Moved to nested object
    firstName?: string;
    lastName?: string;
  };
}
```

Handle migrations at the repository level to maintain backward compatibility.
